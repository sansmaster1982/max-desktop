# MAX Desktop

Неофициальный десктоп-клиент мессенджера MAX (`api.oneme.ru`) на Python и
PySide6. Протокол восстановлен реверс-инжинирингом: бинарный кадр + msgpack,
TLS к `api.oneme.ru:443`. Это форк-клиент в том же смысле, в каком существуют
сторонние клиенты Telegram — он говорит с официальными серверами MAX по их
протоколу.

## Дисклеймер

Неофициальный проект, не связан с MAX и его правообладателями. Сделан для
интероперабельности и исследовательских целей, как сторонние клиенты у других
мессенджеров. Поставляется «как есть», без гарантий; используете на свой риск и
под свою ответственность, в рамках условий использования сервиса и местного
законодательства.

Приватность: клиент ходит только на серверы MAX (`api.oneme.ru`) и, при сбое
системного DNS, на публичные DoH-резолверы (`1.1.1.1` / `8.8.8.8`). Никакой
телеметрии и отправки данных третьим лицам нет. Токен и кэш хранятся локально в
`%APPDATA%\MAX Desktop\` и в репозиторий не попадают.

## Что умеет

- Переписка: текст, история чатов, входящие через server-push, индикатор
  соединения.
- Медиа: отправка фото, видео и файлов; голосовые сообщения (запись через
  QtMultimedia, отправляются как attach `_type=AUDIO`); просмотр фото и
  воспроизведение видео/голоса во встроенном плеере.
- Два способа входа: по SMS (как в оригинальном MAX, с поддержкой 2FA) и по
  готовому auth-токену из веб-версии `web.max.ru`.
- Импорт контактов отдельной кнопкой: файл `.vcf` (vCard) или `.csv`. Каждый
  номер проверяется через `CONTACT_INFO_BY_PHONE`; найденные в MAX сохраняются.
  Плюс добавление по одному номеру.
- Настройки в объёме оригинала: профиль, тёмная/светлая тема и акцент,
  уведомления, режим Enter, активные сессии, версия, выход.

## Скачать готовый exe

Самодостаточный `MAX Desktop.exe` (Windows x64, ~63 МБ) — в разделе
**[Releases](../../releases)**. Python и зависимости не нужны: скачать и
запустить. Внутри упакованы Qt и мультимедийные плагины (FFmpeg), поэтому видео
и голосовые работают из коробки. Первый запуск чуть дольше — onefile
распаковывается во временную папку.

Пересобрать самому: `build_exe.bat` (ставит зависимости и PyInstaller, кладёт
свежий exe рядом). Сборка проверяется режимом `MAX Desktop.exe --selftest` — он
офлайн грузит все модули и QtMultimedia и завершается с кодом 0.

## Установка (запуск из исходников)

Python 3.11+.

```
pip install -r requirements.txt
```

`PySide6` тянет `PySide6-Essentials` и `PySide6-Addons`. Addons содержит
QtMultimedia с FFmpeg-бэкендом — без него запись голосовых и встроенное
воспроизведение видео отключаются (фото и отправка файлов продолжают работать,
видео открывается во внешнем плеере).

## Запуск

```
python run.py
```

или

```
python -m maxclient
```

## Вход

По SMS: номер в формате `+79991234567`, затем код из SMS, затем — если включён —
пароль 2FA. INIT в этом режиме идёт с `deviceType=ANDROID`.

По токену: открыть `web.max.ru`, войти, затем `F12` → Console и выполнить —
команда покажет ключи `localStorage` с длинными значениями, среди них auth-token
(~700 символов):

```js
Object.entries(localStorage).filter(([k,v])=>v.length>40).forEach(([k,v])=>console.log(k,'=>',v))
```

Скопировать значение токена и вставить в «По токену → Войти по токену». Веб-токен
сервер принимает только при `deviceType=WEB` — этот режим переключает тип
устройства сам. Токен живёт недолго; если вход перестал проходить — возьмите
свежий тем же способом.

Токен после входа лежит локально в `%APPDATA%\MAX Desktop\max_auth_token.txt`.
При следующем запуске сессия восстанавливается автоматически.

## Архитектура

```
maxclient/
  protocol/        транспорт и протокол (не зависит от GUI)
    codec.py       сборка/разбор 10-байтного кадра, msgpack
    raw_parsers.py байтовые парсеры полей (LOGIN/HISTORY часто не декодятся целиком)
    client.py      MaxClient: TLS, читающий поток, матч по seq, push, реконнект
    uploader.py    двухступенчатая загрузка медиа -> attach для MSG_SEND
    opcodes.py     карта опкодов
  core/            оркестрация (не зависит от GUI)
    models.py      Chat, Message, Contact, Attach, Profile
    config.py      пути, настройки, хранилище токена
    store.py       кэш SQLite (чаты, контакты, история)
    service.py     MaxService: связывает client + store + session
  ui/              PySide6
    login_window.py / main_window.py / chat_view.py
    contacts_dialog.py / settings_dialog.py / media.py
    widgets/       avatar, chat_list_item, message_bubble, общие утилиты
    theme.py       палитры и QSS, тёмная/светлая
    async_runner.py  блокирующие вызовы сервиса вне GUI-треда
run.py             точка входа
```

Сетевые методы блокирующие и вызываются из пула потоков (`run_async`), результат
возвращается в GUI-тред через Qt-сигналы. Server-push приходит в читающем потоке
и проксируется в UI тем же механизмом.

## Сжатие кадров (важно)

Тело кадра MAX СЖАТО. Флаг — старший байт поля длины (`header[6]`); младшие 3
байта — сжатая длина. Значения: `0` — без сжатия, `0xFF` — zstd, `>0` — LZ4
(raw block, библиотека `net.jpountz.lz4` в APK; см. `lp.java:172-191` в
декомпиле). Клиент распаковывает тело по флагу ДО msgpack —
`codec.decompress_body` / `codec.lz4_block_decompress` (потоковый LZ4 на чистом
Python, размер знать не нужно) и `zstandard` для zstd. Без этого шага история,
текст сообщений, имена и сессии читались «грязно» (символы-замены, утёкшие имена
полей) — теперь декодируются чисто.

## Что проверено и что выведено по аналогии

Проверено вживую: TLS-подключение к `api.oneme.ru:443`, INIT/LOGIN/SEND/HISTORY
проходят (`cmd=1`); LZ4-распаковка тела даёт чистый msgpack (сессии, история,
профиль). Подтверждает host, формат кадра, флаг сжатия, msgpack, seq.

Проверено офлайн: round-trip кадра, байтовые парсеры на синтетике, разбор
LOGIN-чатов и push, извлечение upload-токена/URL, сборка UI без дисплея
(`tests/test_protocol.py`, `tests/smoke_test.py`).

Выведено по декомпилу и требует уточнения на живом сервере: точные имена полей
в ответе upload-опкодов и в HTTP-ответе upload-сервера (`uploader.py` перебирает
известные варианты ключей — `photoToken`/`token`/`fileId`/`videoId` и вложенные
`result.*`). Поля `cid`, `wave` (форма волны для голосовых) не отправляются.
При смене формата сервером правки локализованы в `uploader.py`.

## Версии протокола

`maxclient/protocol/client.py`: `PROTO_VER=10`, `DEFAULT_APP_VERSION=26.15.0`,
host `api.oneme.ru`. При мажорном обновлении официального приложения может
потребоваться поднять `app_version` (поле есть в настройках).

## Диагностика

`[Errno 11001] getaddrinfo failed` — системный DNS не разрешил `api.oneme.ru`
(фильтрующий провайдер, сбой VPN, временный сбой DNS). Клиент это обходит сам:
при ошибке системного DNS адрес резолвится через DNS-over-HTTPS (Cloudflare
`1.1.1.1`, Google `8.8.8.8` — заданы по IP, поэтому не зависят от системного
DNS), после чего соединение идёт по IP с правильным SNI и проверкой сертификата
на `api.oneme.ru`. Если и DoH недоступен — проверьте интернет, DNS и VPN.

## Ограничения

История и текст теперь декодируются чисто (после распаковки кадров). Группы
видны как обычные чаты без специфики (права, участники). Реакции, ответы и
пересылка не реализованы. Эхо собственных сообщений: при push-копии без
распознанного `messageId` возможен дубль — дедуп по `(chatId, server_id)`.
Точечное завершение чужой сессии — best-effort по `time` сессии (отдельного id
у сессии в протоколе нет); гарантированно выкидывает устройства смена 2FA.
